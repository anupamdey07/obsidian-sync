In FabricOS, **Biological CI/CD** is the automated "Continual Evolution" pipeline that manages the transition from fluid, creative intelligence to deterministic robotic action [1, 2]. This process ensures that "revolutionary" skills generated by the Left Brain survive a rigorous evolutionary filter before being promoted to the immutable hardware layer [2, 3].

### **1. The Foundation: The Command Manifest**
The heart of this system is a **pre-generated, signed, and immutable registry** of every action the Left Brain is permitted to propose [4]. 
*   **Trigger Point:** System 2 (SLM) is trained on a pre-compiled library of command sheets, packages, and metadata [5].
*   **Risk Tier Evaluation:** Every request is categorized to determine the depth of the CI/CD check [4]:
    *   **Risk Tier 0 (Auto-Approved):** System 2 does not wake up; action is immediate.
    *   **Risk Tier 1 (Fast-Path):** A simple match against the manifest for instant verification.
    *   **Risk Tier 2 (Deep Eval):** Required for any action that mutates configurations, installs packages, or writes to the Right Brain filesystem.
    *   **Unknown Commands:** These result in a **hard block** with no negotiation permitted [4].

### **2. The Three-Stage Maturation Pipeline**
A new capability must pass through three distinct phases before it is permanently added to the system's repertoire [1, 2]:

*   **Phase I: Left-Brain CI (Creative Sandbox):** This is a rapid, iterative testing ground where the LLM is allowed to prototype new Python or Docker drivers when new hardware connects [1]. In this stage, "hallucinations" are permitted as part of the creative prototyping process [1].
*   **Phase II: Right-Brain CI (Deterministic Linter):** This is the strict gatekeeper. To pass, experimental code must satisfy three requirements [1, 2]:
    1.  **Formal Schema Check:** It must match strict Zenoh-RPC naming conventions.
    2.  **Resource Sandboxing:** It must pass a simulation that tests memory and CPU limits to prevent Out-Of-Memory (OOM) crashes.
    3.  **Property-Based Testing:** The code is fuzz-tested using frameworks like **Hypothesis** to evaluate its performance against edge-case physical inputs [2].
*   **Phase III: Promotion:** Once the code survives the linter, it is signed by **System 2** and locked into the **immutable `/fabric/bin/` path** [1, 6]. This permanently upskills the system, moving the knowledge from the fluid Left Brain to the deterministic Right Brain [1, 2].

### **3. The Evaluation Loop: Structured JSON Verdicts**
To prevent "context rot" or endless rephrasing loops, System 2 communicates with the Left Brain exclusively through **structured JSON enums** rather than free-text prose [7-9]. This schema includes a `retry_permitted` field; if set to false, the Left Brain must throw a `RejectedActionError` immediately rather than attempting to rephrase the failed intent [9].
